Imaginate un proyecto grande con algunas docenas de APIs, y cientos de conversiones Model → DTO... Hacer esto manualmente puede rápidamente crear mucho código boilerplate, ser propenso a errores, y consumir mucho tiempo. Por suerte para nosotros, hay múltiples [frameworks de mapeo de objetos](https://www.baeldung.com/java-performance-mapping-frameworks).

## ¿Cómo funciona un Mapper?

### MapStruct Spring Extensions

**Para aplicaciones Spring Boot en Java/Kotlin**, recomiendo [MapStruct Spring Extensions](https://mapstruct.org/documentation/spring-extensions/reference/html/), un procesador de anotaciones diseñado para complementar el proyecto core de [MapStruct](https://mapstruct.org/) con características específicas del Spring Framework.

MapStruct funciona a través de **generación de código en tiempo de compilación**. Cuando construís tu proyecto, el procesador de anotaciones de MapStruct escanea tus interfaces de mapper y genera clases de implementación concretas con llamadas plain getter/setter (el mismo código boilerplate que escribirías a mano, pero automatizado).

Este enfoque ofrece varios beneficios:

- **Rendimiento**: Sin overhead en tiempo de ejecución ya que los mapeos se resuelven durante la compilación
- **Seguridad de tipos**: Los errores de mapeo se capturan en tiempo de compilación, no en producción
- **Debuggeable**: El código generado es legible y se puede debuggear paso a paso

### ModelMapper

**Para aplicaciones Groovy Spring Boot**, [la naturaleza dinámica de Groovy entra en conflicto con la generación de código en tiempo de compilación de MapStruct](https://stackoverflow.com/questions/72197127/mapstruct-with-groovy). Por eso recomiendo [ModelMapper](https://modelmapper.org/).

En lugar de generar código en tiempo de compilación, inspecciona los objetos de origen y destino en tiempo de ejecución usando [Java's Reflection API](https://www.baeldung.com/java-reflection) para descubrir campos, compararlos por nombre/convención, y copiar valores.

En tiempo de ejecución, ModelMapper:

1. Inspecciona la clase de origen para encontrar: id, title, releaseYear, ...
2. Inspecciona la clase de destino para encontrar: id, title, releaseYear, ...
3. Compara propiedades por convención de nombres
4. Crea el mapeo y copia valores via reflection

Este enfoque ofrece diferentes compensaciones:

- **Flexibilidad**: Funciona sin problemas con el tipado dinámico y metaprogramming de Groovy
- **Menos configuración**: Automáticamente compara propiedades por convención sin mapeo de campos explícito
- **Costo en tiempo de ejecución**: La reflection es más lenta que las llamadas de método directo, aunque ModelMapper cachea mapeos para minimizar overhead
