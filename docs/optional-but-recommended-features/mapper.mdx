Imagine a big project with a few dozen APIs, and hundreds of Model → DTO conversions... Doing this manually can quickly create much boilerplate code, be prone to error, and consume a lot of time. Luckily for us, there are multiple [object mapping frameworks](https://www.baeldung.com/java-performance-mapping-frameworks).

## How Does A Mapper Work?

### MapStruct Spring Extensions

**For Java/Kotlin Spring Boot apps**, I recommend [MapStruct Spring Extensions](https://mapstruct.org/documentation/spring-extensions/reference/html/), an annotation processor designed to complement the core [MapStruct project](https://mapstruct.org/) with features specific to the Spring Framework.

MapStruct works through **compile-time code generation**. When you build your project, the MapStruct annotation processor scans your mapper interfaces and generates concrete implementation classes with plain getter/setter calls (the same boilerplate code you would write by hand, but automated).

This approach offers several advantages:

- **Performance**: No runtime overhead since mappings are resolved during compilation
- **Type safety**: Mapping errors are caught at compile time, not in production
- **Debuggable**: Generated code is readable and can be stepped through in a debugger

### ModelMapper

**For Groovy Spring Boot apps**, [Groovy's dynamic nature conflicts with MapStruct's compile-time code generation](https://stackoverflow.com/questions/72197127/mapstruct-with-groovy). That's why I recommend [ModelMapper](https://modelmapper.org/).

Instead of generating code at compile time, it inspects the source and destination objects at runtime using [Java's Reflection API](https://www.baeldung.com/java-reflection) to discover fields, match them by name/convention, and copy values.

At runtime, ModelMapper:

1. Inspects the source class to find: `id`, `title`, `releaseYear`, ...
2. Inspects the target class to find: `id`, `title`, `releaseYear`, ...
3. Matches properties by naming convention
4. Creates mapping and copies values via reflection

This approach offers different trade-offs:

- **Flexibility**: Works seamlessly with Groovy's dynamic typing and metaprogramming
- **Less configuration**: Automatically matches properties by convention without explicit field mapping
- **Runtime cost**: Reflection is slower than direct method calls, though ModelMapper caches mappings to minimize overhead
